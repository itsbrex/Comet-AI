## Highlights

Critical fix for macOS app launch - the app now works with ad-hoc code signing without requiring an Apple Developer account.

### What's Fixed

| Issue | Status |
|-------|--------|
| App not launching on macOS | ✅ Fixed |
| Restricted entitlements error | ✅ Fixed |
| AMFI signature validation failure | ✅ Fixed |

---

## The Problem

Previously, the app contained **restricted entitlements** that require an Apple Developer certificate:

- `com.apple.security.cs.allow-jit`
- `com.apple.security.cs.allow-unsigned-executable-memory`
- `com.apple.security.cs.disable-library-validation`

When the app was built with ad-hoc signing (no Apple Developer certificate), the macOS kernel blocked the app from launching:

```
AMFI: '/Applications/Comet-AI.app/Contents/MacOS/Comet-AI' is adhoc signed.
amfid: The file is adhoc signed but contains restricted entitlements
```

## The Solution

Removed the restricted entitlements from `entitlements.mac.local.plist`. The app now only uses entitlements allowed with ad-hoc signing:

```xml
<dict>
  <key>com.apple.security.network.client</key>
  <true/>
  <key>com.apple.security.network.server</key>
  <true/>
  <key>com.apple.security.files.user-selected.read-write</key>
  <true/>
  <key>com.apple.security.automation.apple-events</key>
  <true/>
</dict>
```

## Note on Apple Developer Certificate

If you have an Apple Developer account and want maximum functionality, you can add these entitlements back when signing with a valid certificate:

```xml
<key>com.apple.security.cs.allow-jit</key>
<true/>
<key>com.apple.security.cs.allow-unsigned-executable-memory</key>
<true/>
<key>com.apple.security.cs.disable-library-validation</key>
<true/>
```

For ad-hoc/local development, the simplified entitlements work perfectly.
